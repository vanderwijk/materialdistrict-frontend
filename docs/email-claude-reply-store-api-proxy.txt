=== Proxy: cart-only of generiek? ===

**Generiek.** De catch-all route mapped:

  /api/store-cart/{…path}  →  {WP_API_URL}/wc/store/v1/{…path}

Dus POST /checkout en GET /order/{id} lopen er vanzelf doorheen — geen extra
Next-routes nodig van mijn kant. Voorbeelden:

  POST /api/store-cart/checkout
  GET  /api/store-cart/order/123?key=…&billing_email=…

De routenaam `/api/store-cart` is historisch (eerst alleen mand), maar de
implementatie dekt de hele Store API. Jij kunt `cartFetch` uitbreiden of een
gedeelde `storeApiFetch` maken met dezelfde base `/api/store-cart` — dat is
mijn voorkeur (één client, één proxy).

Wat de proxy doet per request:
- Forward: Content-Type, Cart-Token, Nonce, X-WC-Store-API-Nonce
- Voegt toe: Authorization: Bearer {JWT} wanneer md_auth_token-cookie aanwezig
- Response: Cart-Token / Nonce headers terug naar de browser
- Methodes: GET, POST, PUT, PATCH, DELETE

Order confirmation heeft meestal geen JWT nodig (order_key + billing_email als
credential). Cart-Token wél relevant voor POST checkout zolang dezelfde sessie.
